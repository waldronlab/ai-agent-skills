# Bioconductor Package Development Standards

Standards for **developing** Bioconductor/CRAN packages. These are referenced by package-specific instruction files but maintained centrally.

**Prerequisites**: Package developers should also understand [bioconductor-usage.md](bioconductor-usage.md) for working with dependencies and understanding how users will interact with their package.

**For package-specific patterns and conventions**, see the package's `.github/instructions/` directory.

## Overview

This document covers:
- **Code style** - Formatting, naming, spacing per Bioconductor standards
- **Function development** - Best practices, prohibited patterns, efficiency requirements
- **S4 classes** - Structure, file organization, validation, reuse mandate
- **Documentation** - roxygen2 requirements, examples, vignettes
- **Testing** - Framework, organization, coverage requirements
- **Dependencies** - Import requirements, restrictions on data sources
- **Parallel computing** - BiocParallel requirements
- **R CMD check** - Pre-submission checklist, BiocCheck requirements
- **Version numbering** - Pre-Bioc and Bioconductor versioning
- **Package structure** - Required/recommended files, file naming

**Based on**: [Official Bioconductor R Code Guidelines](https://contributions.bioconductor.org/r-code.html)

## Documentation Standards

### Required roxygen2 Tags for Exported Functions

All exported functions MUST include:
- `@title` (or first line serves as title)
- `@description`
- `@param` for each parameter
- `@return` describing the return value
- `@examples` with runnable examples
- `@export` to make the function available to users

### Example Guidelines

**Use `\donttest{}`**:
- For examples that access remote resources
- For examples that may hit rate limits
- For examples that take >5 seconds to run

**Use `\dontrun{}`**:
- Rarely - only for examples that should never run automatically
- Better: provide a working local alternative

**Best practice**:
- Provide both remote and local examples when applicable
- Always include at least one runnable example
- Examples should be self-contained and reproducible

### Documentation Building

Regenerate documentation after any roxygen changes:
```r
roxygen2::roxygenize()
```

## Testing Standards

### Framework
Use `testthat` for unit tests.

### Test Organization
```
tests/
└── testthat/
    ├── test-function-name.R
    └── test-another-function.R
```

### Test Data
- **Location**: `inst/extdata/` for package data, or `tests/testthat/fixtures/`
- **Principle**: Tests should use small, local files when possible
- **Remote resources**: Avoid if possible; use local test data instead

### Test Coverage
- Cover exported functions
- Test invalid input and error handling
- Test edge cases

### Running Tests
```r
# All tests
devtools::test()

# Specific test file
testthat::test_file("tests/testthat/test-name.R")
```

## R CMD check Requirements

### Pre-Submit Checklist

Before any commit or PR:

1. **Build the package**:
   ```bash
   R CMD build .
   ```

2. **Run R CMD check** (must pass with no errors, warnings, or notes):
   ```bash
   R CMD check PackageName_*.tar.gz
   ```

3. **Run BiocCheck** (for Bioconductor packages):
   ```bash
   R -e "BiocCheck::BiocCheck('PackageName_*.tar.gz')"
   ```

4. **Test examples**:
   ```r
   devtools::run_examples()
   ```

5. **Build vignettes** (if applicable):
   ```r
   devtools::build_vignettes()
   ```

### Common Check Issues

- Missing documentation
- Non-ASCII characters
- Long-running examples without `\donttest{}`
- Missing imports in NAMESPACE
- Unused dependencies in DESCRIPTION

## Version Numbering

### Pre-Bioconductor (Development)
- Use `0.99.x` version numbering
- Increment `x` for each significant change

### After Bioconductor Acceptance
- Use semantic versioning: `x.y.z`
- **x**: Major version (rarely changes, Bioconductor manages)
- **y**: Minor version (increment for new features)
- **z**: Patch version (increment for bug fixes)

### Version Bumping Process
1. Update version in `DESCRIPTION`
2. Commit with message: `chore: bump version to x.y.z`
3. Create version-specific git tag (optional)

## Package Structure Requirements

### Required Files
- `DESCRIPTION` - Package metadata
- `NAMESPACE` - Exported/imported functions (auto-generated by roxygen)
- `R/` - R source code
- `man/` - Documentation (auto-generated by roxygen)

### Recommended Files
- `README.md` - Package overview and installation
- `NEWS.md` - Version history and changes
- `tests/testthat/` - Unit tests
- `vignettes/` - Long-form documentation
- `inst/extdata/` - Example/test data

### Build Configuration
`.Rbuildignore` should exclude development files:
```
^\.github$
^\.git$
^\.Rproj\.user$
^.*\.Rproj$
^LICENSE\.md$
```

## Code Style

**Reference**: [Bioconductor R Code Guidelines](https://contributions.bioconductor.org/r-code.html)

### Formatting Requirements

**Indentation and Line Length**:
- Use **4 spaces** for indentation (never tabs)
- Maximum line length: **80 characters**

**Naming Conventions**:
- **Functions**: `camelCase` (initial lowercase) - e.g., `calculateDiversity()`
- **Classes**: `CamelCaps` (initial uppercase) - e.g., `ExperimentData`
- **Variables**: `camelCase`
- **Non-exported functions**: prefix with `.` (dot) - e.g., `.internalHelper()`
- **Avoid** using `.` in function names (conflicts with S3 method dispatch)

**Spacing Standards**:
- Space after commas: `a, b, c`
- **No space** around `=` in named function arguments: `func(a=1, b=2)`
- Space around binary operators: `a == b`, `x + y`
- Use `<-` for variable assignment (except function arguments where `=` is used)

**Comments**:
- Use `##` for full-line comments at same indentation level as code
- No commented-out code chunks in published packages
- Avoid TODO comments in production code

### General Principles
- Use meaningful, descriptive names
- Keep functions small and focused (avoid functions exceeding one screen length)
- Break large functions into smaller helper functions
- Validate inputs early with `stopifnot()` or explicit checks
- Provide informative error messages

### Function Development Requirements

**Prohibited in package code**:
- ❌ `set.seed()` - Never in internal code
- ❌ `browser()` - No debugging calls
- ❌ `<<-` - Avoid super-assignment operator
- ❌ Nesting helper functions within main functions

**Best Practices**:
- Provide default values for arguments
- Validate all arguments
- Use descriptive, well-documented parameter names
- Return explicit values

### Internal Helpers
- Use `@noRd` for internal helpers (not exported)
- Prefix with `.` for non-exported functions
- Consider naming patterns like `.validate*`, `.check*`, `.helper*`

## End-User Communication

Use appropriate functions for different message types:

- **`message()`** - For diagnostic or progress information
  ```r
  message("Processing sample ", i, " of ", n)
  ```

- **`warning()`** - For unusual situations that aren't errors
  ```r
  warning("Low read depth detected for sample ", sampleId)
  ```

- **`stop()`** - For error conditions
  ```r
  stop("Input must be a SummarizedExperiment object")
  ```

- **`cat()` or `print()`** - **Only** in `show()` methods, not in regular functions

**Avoid `paste0()`** except when collapsing multiple values into one string.

## Code Efficiency and Best Practices

### Required Patterns

**Use vectorized operations**:
- ✅ `vapply()` instead of `sapply()` (type-safe)
- ✅ `lapply()` for list operations
- ✅ `apply()` family instead of `for` loops when possible

**Safe indexing**:
- ✅ `seq_len(n)` instead of `1:n`
- ✅ `seq_along(x)` instead of `1:length(x)`
- Why: Handles edge cases (n=0) correctly

**Type checking**:
- ✅ `is(x, "ClassName")` instead of `class(x) == "ClassName"`
- Why: Works correctly with inheritance

**System calls**:
- ✅ `system2()` instead of `system()`

**Boolean values**:
- ✅ `TRUE` and `FALSE` (never `T` or `F`)

**Pre-allocation**:
```r
# Good: pre-allocate
result <- vector("list", n)
for (i in seq_len(n)) {
  result[[i]] <- compute(data[[i]])
}

# Bad: copy-and-append (quadratic complexity!)
result <- list()
for (i in seq_len(n)) {
  result <- c(result, compute(data[[i]]))  # ❌ Don't do this
}
```

### Vectorization Priority

Replace iterative approaches with vectorized operations whenever possible. When loops are necessary, "hoist" calculations outside the loop if they don't depend on loop variables.

### Edge Case Handling

Write robust code that handles:
- Zero-length vectors
- `NA` values
- Empty inputs

Use `identical()` or `all.equal()` for correctness verification.

### Cyclomatic Complexity

Keep control-flow complexity low by:
- Reducing nested conditionals
- Using simpler internal representations
- Breaking complex logic into helper functions

## S4 Classes

**Bioconductor strongly prefers S4 over S3 classes.**

### Class Reuse Mandate

**Before creating new classes**: Consult existing Bioconductor classes and consider:
- Can you extend an existing class?
- Can you compose with established structures (SummarizedExperiment, MultiAssayExperiment, etc.)?
- Is a new class truly necessary?

Do not implement new classes without investigating reuse options first.

### Essential Components

Every S4 class MUST have:

1. **Constructor function** for object creation
   ```r
   #' @export
   MyClass <- function(data, metadata) {
     new("MyClass", data=data, metadata=metadata)
   }
   ```

2. **`show()` method** for appropriate display
   ```r
   #' @export
   setMethod("show", "MyClass", function(object) {
     cat("MyClass object with", nrow(object@data), "features\n")
   })
   ```

3. **Accessor functions** instead of direct `@` slot access
   ```r
   #' @export
   getData <- function(x) x@data  # Lightweight name-mangling with 2-3 letter prefix
   ```

4. **Validity method** for object validation
   ```r
   setValidity("MyClass", function(object) {
     if (ncol(object@data) == 0)
       return("data cannot be empty")
     TRUE
   })
   ```

### File Organization

**Strict organization requirements**:
- **All class definitions** → `R/AllClasses.R`
- **All generic definitions** → `R/AllGenerics.R`
- **Methods** → Files named by generic: `R/show-methods.R`, `R/methods-MyClass.R`

If source order matters, include `Collate:` field in DESCRIPTION to control loading order.

### Class Definition Example
```r
#' @title ClassName
#' @description Brief description of the class
#' @slot data Description of data slot
#' @slot metadata Description of metadata slot
#' @exportClass ClassName
setClass("ClassName",
  slots = c(
    data = "matrix",
    metadata = "list"
  )
)
```

### Methods and Generics

**Reuse existing generics**: Check `BiocGenerics` for generics before creating new ones. Only create new generics when behavior doesn't align with existing generic contracts.

**Method creation restriction**:
- ❌ Do NOT create methods for external classes (classes from other packages)
- ✅ Only create methods for classes exported within your package
- Why: Prevents method name collisions and environment pollution

## Dependencies and Namespace Management

### DESCRIPTION Fields

**Imports**: Packages required for your package to work
```
Imports:
    dplyr,
    SummarizedExperiment,
    BiocGenerics
```

**Suggests**: Packages needed for vignettes, tests, or optional features
```
Suggests:
    testthat,
    knitr,
    rmarkdown,
    BiocStyle
```

**Depends**: Packages that must be attached (use sparingly, prefer Imports)

### Critical Requirements

**All dependencies MUST be on CRAN or Bioconductor**:
- ❌ Bioconductor does NOT recognize `Remotes:` field
- ❌ Cannot depend on GitHub, GitLab, or other non-standard repositories
- ✅ Submit dependencies to CRAN/Bioconductor first if needed

**Import enumeration**:
- **Import ALL symbols** from non-base packages
- Exception: Default packages (base, graphics, stats, methods, utils, grDevices) don't need explicit imports
- Fully enumerate imports in NAMESPACE

**Export enumeration**:
- Export all symbols useful to end-users
- Fully enumerate exports (don't use `exportPattern()`)

### Using Functions from Dependencies

In code:
```r
# Good: explicit namespace
dplyr::filter(data, condition)

# Acceptable if imported
filter(data, condition)  # Only if @importFrom dplyr filter
```

In NAMESPACE (via roxygen):
```r
#' @importFrom dplyr filter mutate select
#' @importFrom BiocGenerics start end
```

### File System Restrictions

**CRITICAL: Do NOT install anything on a user's system!**

❌ **Never write to**:
- Home directory
- Working directory
- Package installation directory

✅ **Approved locations**:
- `BiocFileCache::BiocFileCache()` - For cached data
- `tools::R_user_dir(package, "cache")` - For cache files
- `tempdir()` / `tempfile()` - For temporary files only

## Parallel Computing

**Use the `BiocParallel` package** for parallel operations.

### Requirements

1. **Use `lapply()`-style iteration** instead of explicit loops
   ```r
   # Good
   results <- bplapply(data_list, processFunction, BPPARAM=param)

   # Avoid explicit for loops with parallel backends
   ```

2. **Function must be self-contained**
   ```r
   # Function passed to bplapply() must not depend on external state
   processFunction <- function(x) {
     # All needed variables passed as arguments
     # No reliance on parent environment
   }
   ```

3. **Allow user-specified backend**
   ```r
   myFunction <- function(data, BPPARAM=BiocParallel::SerialParam()) {
     BiocParallel::bplapply(data, worker, BPPARAM=BPPARAM)
   }
   ```

4. **Set minimal cores as default**
   - Default to 1-2 cores maximum
   - Let users scale up via `BiocParallel::register()`

### Example
```r
library(BiocParallel)

processData <- function(data, cores=1) {
  param <- MulticoreParam(workers=cores)
  bplapply(data, analyzeRow, BPPARAM=param)
}
```

## Data Sources and External Data

### Approved Data Sources

**REQUIRED: Use `ExperimentHub` and `AnnotationHub`** for external data:
- ✅ ExperimentHub for experimental data packages
- ✅ AnnotationHub for annotation resources
- ✅ Trusted public databases (NCBI, EBI, etc.)

**DISCOURAGED**:
- ❌ GitHub raw files
- ❌ Dropbox links
- ❌ Personal websites
- ❌ Unsanctioned third-party providers

### Rationale
Data must come from curated, version-controlled, and persistent sources. ExperimentHub/AnnotationHub provide:
- Version control
- Persistent identifiers
- Bioconductor integration
- Caching infrastructure

### Example
```r
# Good: Using ExperimentHub
library(ExperimentHub)
eh <- ExperimentHub()
mydata <- eh[["EH1234"]]

# Bad: Direct download from GitHub
# ❌ Don't do this
download.file("https://github.com/user/repo/raw/main/data.rds", ...)
```

## Bioconductor Submission

### First-Time Submission
1. Ensure version is 0.99.0 or higher
2. Pass R CMD check with no errors, warnings, or notes
3. Pass BiocCheck with no errors
4. Have at least one vignette
5. Submit via Bioconductor GitHub submission tracker

### Continuous Integration
- Set up GitHub Actions for R CMD check
- Run BiocCheck in CI
- Build and check vignettes

## File Naming Conventions

### R Source Files
- Extension: `.R`
- Method files: `methods-[classname].R` or `[generic]-methods.R`
  - Example: `methods-ExperimentData.R` or `show-methods.R`
- Class organization:
  - `R/AllClasses.R` - All class definitions
  - `R/AllGenerics.R` - All generic function definitions

### Documentation Files
- Extension: `.Rd` (auto-generated by roxygen2)
